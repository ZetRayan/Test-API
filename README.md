# API Организационной структуры

REST API для управления иерархией подразделений и сотрудниками компании. Разработано в рамках тестового задания. Включает строгую валидацию данных, защиту древовидной структуры от зацикливания и каскадного уничтожения сотрудников.

## Технологический стек

* **Язык:** Python 3.11
* **Фреймворк:** FastAPI
* **База данных:** PostgreSQL 15
* **ORM & Миграции:** SQLAlchemy + Alembic
* **Тестирование:** Pytest + HTTPX (TestClient)
* **Инфраструктура:** Docker, Docker Compose

## Архитектура проекта

Проект разделен на логические слои (изолированная ответственность):
* **main.py** — Точка сборки и инициализации приложения.
* **database.py** — Настройка движка SQLAlchemy и управление сессиями подключения.
* **models.py** — Описание таблиц базы данных (слой ORM).
* **schemas.py** — Валидация входящих и исходящих данных через Pydantic.
* **routers/** — Обработка HTTP-запросов и маршрутизация (отдельно для отделов и сотрудников).
* **tests/** — Автотесты (вынесены в отдельный изолированный пакет).

## Запуск проекта

Сборка и запуск контейнеров (база данных + API приложение). Миграции Alembic применяются автоматически при старте сервиса.

```Bash
docker-compose up -d --build
```

* Swagger UI (интерактивное тестирование): http://localhost:8000/docs

## Тестирование

Бизнес-логика полностью покрыта 24 автотестами. Тесты изолированы от основной базы данных и используют базу SQLite в оперативной памяти (:memory:), которая уничтожается сразу после завершения проверок.

Для запуска тестов внутри работающего контейнера выполните:

```Bash
docker-compose exec api python -m pytest -v
```
## Полная зачистка базы данных

Если необходимо полностью уничтожить локальную базу данных, включая созданные таблицы и сохраненные данные, выполните:

```Bash
docker-compose down -v
```
При следующем запуске через `up` база поднимется чистой с автоматическим пересозданием структуры.

## Особенности реализации бизнес-логики

1. **Защита от циклических зависимостей (Уроборос):** В методе PATCH `/departments/{id}` реализован алгоритм обхода дерева вверх к корню. Это исключает возможность назначить подразделение родителем самого себя или переместить его внутрь собственного дочернего узла.
2. **Лимиты рекурсии:** В методе GET `/departments/{id}` установлен жесткий лимит глубины обхода дерева (depth от 1 до 5) для предотвращения избыточной нагрузки на БД. По умолчанию возвращается глубина 1.
3. **Безопасная эвакуация при удалении:** При DELETE `/departments/{id}` в режиме `reassign` реализована проверка «черной дыры». Система запрещает переводить сотрудников во вложенные отделы удаляемого подразделения, так как иначе они были бы уничтожены каскадным удалением базы данных вместе с этим подотделом.
